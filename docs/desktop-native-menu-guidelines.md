# 小店AI客服桌面端原生菜单设计规范

> 本文定位：Mac 原生菜单、Windows 三条杠菜单、托盘和快捷键的 UI source of truth。上级规范：[ARCHITECTURE.md](../ARCHITECTURE.md)。

这份规范定义桌面端全局操作入口。后续涉及 Mac 菜单栏、Windows 三条杠菜单、托盘菜单、窗口内快捷入口和快捷键时，都以本文为准。

## 设计原则

桌面端的全局命令必须遵守四条原则：

| 原则 | 规则 |
| --- | --- |
| 一套命令 | Mac、Windows、托盘和快捷键必须调用同一套命令定义，不能各写一份逻辑 |
| 平台同构 | Mac 和 Windows 的菜单层级、命名、顺序、状态必须一一对应 |
| 平台原生 | Mac 使用屏幕顶部系统菜单栏；Windows 使用窗口内小型三条杠按钮展开菜单 |
| 去顶部栏 | 主窗口不再保留当前 64px 顶部状态/操作栏，全局操作迁移到原生菜单和 Windows 三条杠菜单 |

这里的“去顶部栏”只指当前主窗口内部的全局 Top Bar，不代表删除页面内的状态、二级导航或局部工具栏。状态和上下文可以留在侧边栏、Context Bar、详情面板或页面局部区域。

## 平台映射

### Mac

Mac 使用系统顶部菜单栏承载全局命令。

要求：

- 菜单出现在屏幕最顶部，由 Electron `Menu.setApplicationMenu()` 设置。
- 顶级业务菜单顺序固定为：`设置`、`工作台设定`、`Bot`、`悬浮窗`、`API`。
- 主窗口必须隐藏 macOS 原生标题文字，只保留红黄绿窗口按钮；窗口标题不能出现在按钮右侧。
- macOS 必需的应用菜单、编辑菜单、窗口菜单可以保留，但业务命令必须集中在上述五个顶级业务菜单内。
- 菜单项的启用、禁用、勾选状态必须反映当前运行状态。
- 关闭主窗口只是隐藏窗口，彻底退出必须使用明确的退出命令并二次确认。

### Windows

Windows 不使用屏幕顶部系统菜单栏，也不显示一整条传统菜单栏。Windows 使用一个小型三条杠按钮作为全局菜单入口。

要求：

- 三条杠按钮必须是小型图标按钮，不形成新的全宽顶部栏。
- 推荐位置是主窗口侧边栏顶部品牌区附近，或系统设置入口旁边。
- 点击后展开的菜单层级、顺序、命名、状态和 Mac 顶部菜单保持一致。
- 展开菜单可以是弹出菜单、下拉菜单或轻量命令面板，但不能做成新的页面。
- Windows 仍可保留托盘菜单，但托盘菜单只放运行和恢复入口，不能成为唯一配置入口。

### 托盘

托盘菜单是后台运行入口，不是完整设置中心。

允许保留：

- 打开主窗口
- 显示或隐藏悬浮窗
- 开启或暂停 Bot
- 重载客服页
- 检查 AI 服务
- 彻底退出程序

托盘菜单的文案和状态必须来自同一套命令定义。

## 顶级菜单结构

### 设置

用于系统级、低频、跨模块的设置。

| 菜单项 | 规则 |
| --- | --- |
| 打开设置 | 进入主窗口的系统设置视图 |
| 开机启动 | 可勾选，映射当前开机启动配置 |
| 通知设置 | 进入通知/Webhook/系统提醒相关设置 |
| 防休眠 | 可勾选，映射当前防休眠配置 |
| 初始化与修复 | 打开初始化、自检、配置修复入口 |
| 彻底退出程序 | 必须二次确认；会停止 Bot、AI 服务、本机控制服务和悬浮窗 |

### 工作台设定

用于主工作区、页面视图和工作台布局。

| 菜单项 | 规则 |
| --- | --- |
| 打开工作台 | 显示主窗口并聚焦 |
| 工作台 | 切换到微信小店客服页 |
| 知识库 | 切换到规则、动作、AI 风格和判断库视图 |
| 监控 | 切换到状态、日志、异常和 Webhook 队列视图 |
| 设置 | 切换到系统设置视图 |
| 重载客服页 | 重新加载微信小店客服页 |
| 捕捉页面结构 | 执行当前客服页结构捕捉 |

### Bot

用于自动回复接管能力。

| 菜单项 | 规则 |
| --- | --- |
| 开启 Bot | Bot 关闭时可用 |
| 暂停 Bot | Bot 开启时可用 |
| Bot 设定 | 打开回复规则、AI 回复策略、动作规则相关设置 |
| 测试回复 | 打开手动测试入口 |
| 测试规则命中 | 打开规则触发测试入口 |
| 查看最近回复 | 打开最近回复记录或运行监控中的回复详情 |

要求：

- `开启 Bot` 和 `暂停 Bot` 不能同时作为主要动作出现；当前状态决定显示或启用哪一个。
- 菜单项执行后，托盘、悬浮窗、侧边栏状态和 Windows 三条杠菜单必须同步刷新。

### 悬浮窗

用于控制桌面悬浮窗。

| 菜单项 | 规则 |
| --- | --- |
| 开启悬浮窗 | 显示悬浮窗并写入可见状态 |
| 关闭悬浮窗 | 隐藏悬浮窗，不退出后台服务 |
| 展开尺寸 | 切换到展开态 |
| 迷你尺寸 | 切换到最小化态 |
| 置顶显示 | 可勾选，映射 always-on-top 设置 |
| 位置预设 | 可展开为左上、右上、左下、右下等预设 |
| 悬浮窗设定 | 打开悬浮窗详细设置 |

要求：

- 悬浮窗关闭等于隐藏，不等于退出程序。
- 尺寸只能选择预设，不做用户随意拖拽缩放。
- Mac 和 Windows 的尺寸命名保持一致。

### API

用于外部服务、本地服务和连通性检查。

| 菜单项 | 规则 |
| --- | --- |
| 检查 AI 连通性 | 调用当前 AI 健康检查 |
| API 设定 | 打开 AI Key、模型、服务地址等设置 |
| Webhook 设定 | 打开企业微信/Webhook 配置 |
| 测试 Webhook | 执行 Webhook 测试 |
| 判断库登录 | 打开润宇判断库登录窗口 |
| 检查判断库授权 | 执行 Cookie/授权自检 |
| 刷新判断库 | 执行判断库刷新 |

要求：

- 所有检查类动作必须返回明确反馈：成功、失败、失败原因、下一步处理入口。
- API Key、Cookie、Webhook URL 等敏感信息不能出现在菜单文案、日志摘要或系统通知里。

## 主窗口布局规则

当前主窗口内的 Top Bar 需要迁移，不再作为长期固定结构。

迁移后主窗口遵守以下规则：

- 主窗口保留侧边栏作为一级导航和状态摘要。
- 侧边栏展开时从上到下依次显示 Logo、`小店AI客服控制台`、店铺名、悬浮窗状态入口、四个一级入口、工作台详情面板；未登录时店铺名显示 `微信小店`。
- 四个一级入口固定为 `工作台`、`知识库`、`监控`、`设置`，展开态只显示图标和名称，不显示说明文案。
- 侧边栏必须提供独立折叠按钮。Mac 上按钮使用接近红黄绿窗口按钮的尺寸，放在红黄绿右侧安全区，展开和折叠态都不能遮挡系统窗口按钮；Windows 上不能替代三条杠菜单。
- 侧边栏悬浮窗状态区复用悬浮窗最小化态的信息结构：运行灯号、短状态和短摘要；侧边栏内只保留 `展开`、`隐藏` 两个悬浮窗操作。
- 侧边栏不再重复展示 Bot、AI、通知等独立状态条，避免和悬浮窗状态区重复。
- 折叠态只保留 Logo、悬浮窗灯号和四个一级入口图标，隐藏文字说明、悬浮窗按钮和工作台详情面板。
- Mac 全屏后不再保留红黄绿窗口按钮的顶部空白，侧边栏内容应上移。
- 客服页 `BrowserView` 不再因为旧 Top Bar 固定下移 64px。
- 全局高频命令迁移到 Mac 菜单栏、Windows 三条杠菜单、托盘菜单和快捷键。
- 页面内可以保留局部工具栏，例如规则列表的新增、保存、测试按钮。
- 运行状态不再集中堆在顶部栏，应该分散到侧边栏灯号、运行监控页、悬浮窗和异常详情。

## 命令注册规则

当前实现必须使用统一命令模型。每个命令至少包含：

```text
id
label
menu
section
platforms
accelerator
enabled(status)
checked(status)
run(context)
```

约束：

- Mac `Menu`、Windows 三条杠菜单、托盘菜单和快捷键都从同一命令模型派生。
- 不能在不同 UI 里复制业务逻辑。
- 命令执行函数必须复用现有主进程能力，例如 `showMainWindow`、`setBotEnabled`、`reloadKfPage`、`checkAiHealth`、`showFloatingWindow`。
- 状态变化后必须刷新所有菜单表面。

当前 IPC 约定：

| 接口 | 用途 |
| --- | --- |
| `main-get-menu-model` | 返回 Windows 三条杠菜单渲染所需的业务菜单模型 |
| `main-run-menu-command` | 执行菜单命令，Windows 三条杠菜单只传命令 ID |
| `desktop-menu-model` | 主进程向主控台推送最新菜单模型 |
| `main-open-view` | Mac 菜单或托盘触发视图切换时，通知主控台同步页面 |

当前顶级命令 ID 前缀：

| 前缀 | 菜单 |
| --- | --- |
| `settings.*` | 设置 |
| `workbench.*` | 工作台设定 |
| `bot.*` | Bot |
| `floating.*` | 悬浮窗 |
| `api.*` | API |

## 快捷键规则

快捷键是菜单的补充，不替代可见入口。

建议默认：

| 快捷键 | 动作 |
| --- | --- |
| Cmd/Ctrl + 1 | 打开工作台 |
| Cmd/Ctrl + 2 | 显示悬浮窗 |
| Cmd/Ctrl + Shift + B | 开启或暂停 Bot |
| Cmd/Ctrl + R | 重载客服页 |
| Cmd/Ctrl + , | 打开设置 |

Mac 显示 `Cmd`，Windows 显示 `Ctrl`。同一命令在两端保持一致的语义。

## 验收标准

菜单改造完成后必须满足：

- Mac 屏幕顶部出现业务菜单：`设置`、`工作台设定`、`Bot`、`悬浮窗`、`API`。
- Windows 主窗口只有小型三条杠菜单入口，不出现全宽传统菜单栏。
- Windows 三条杠菜单展开后和 Mac 业务菜单一一对应。
- 主窗口旧 Top Bar 被移除，客服页不再被旧 Top Bar 强制下移。
- Bot、悬浮窗、API 检查等状态变化后，Mac 菜单、Windows 菜单、托盘菜单、悬浮窗状态同步更新。
- 彻底退出仍然二次确认。
- 敏感配置不出现在菜单文案或通知摘要中。
- `npm run test:desktop-modules`、`npm run test:lifecycle`、`npm run test:status-ui` 和 `npm run doctor` 能通过，或记录明确的阻塞原因。
