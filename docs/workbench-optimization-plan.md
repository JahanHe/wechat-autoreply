# 小店AI客服新版工作台优化方案

这份方案来自 v0.4.0 前的工作台重构讨论，用于指导后续把主控台从“功能配置集合”继续优化成“对象、状态、动作、反馈”的高频工作台。

## 目标

新版工作台要让客服或运营在 5 秒内知道三件事：

| 问题 | 页面应给出的答案 |
| --- | --- |
| 现在发生了什么 | 当前步骤、登录、脚本、AI、本地服务和异常状态 |
| 哪里需要处理 | 待处理异常、失败回复、判断库失效、Webhook 队列 |
| 我能做什么 | 暂停/开启 Bot、重载客服页、打开悬浮窗、测试规则、修复配置 |

工作台不是营销页，不需要大 Hero、卖点卡片或长文案。它应该是稳定、紧凑、可扫描、可重复使用的操作界面。

## 整体结构

推荐结构：

```text
┌──────────────────────────────────────────────┐
│ Top Bar：当前状态 / 搜索 / 快捷动作 / 异常入口 │
├──────────────┬───────────────────────────────┤
│ Sidebar      │ Context Bar                   │
│ 一级导航      ├───────────────────────────────┤
│              │ Main Workspace                │
│              │ 客服页 / 规则列表 / 日志 / 设置 │
│              ├───────────────────────────────┤
│              │ Status / Action Feedback       │
└──────────────┴───────────────────────────────┘
```

需要查看或编辑对象时，使用右侧详情面板：

```text
┌──────────┬────────────────────────┬──────────┐
│ Sidebar  │ Main Workspace          │ Details  │
│ 模块导航  │ 列表 / 表格 / 客服页       │ 属性/日志 │
└──────────┴────────────────────────┴──────────┘
```

## 信息架构

一级导航保持四个入口：

| 一级入口 | 主对象 | 主动作 |
| --- | --- | --- |
| 客服工作台 | 微信小店客服页和当前会话 | 暂停/开启 Bot、重载页面、捕捉结构 |
| 回复中心 | 规则、动作、AI 风格、判断库 | 新增规则、手动测试、保存配置 |
| 运行监控 | 当前状态、日志、异常、Webhook 队列 | 刷新、检查 AI、测试 Webhook、查看 Trace |
| 系统设置 | 初始化、通知、悬浮窗、开机启动、退出 | 保存设置、自检、彻底退出 |

二级入口只在 Context Bar 或页面内 tabs 出现，不再让 Sidebar 承载大量子菜单。

## 关键页面

### 客服工作台

- 主区保留微信小店客服原网页映射。
- Top Bar 显示当前步骤、AI、本地、脚本、登录和 Bot 状态。
- BrowserView 需要为 Top Bar 留出固定高度，不能覆盖控制入口。
- 高频操作固定可见：打开悬浮窗、暂停/开启 Bot、重载客服页、捕捉页面结构。

### 回复中心

- 主区显示规则列表或规则看板。
- 动作规则、文字规则、图片规则分组展示。
- 选中规则后在右侧详情面板编辑。
- 图片和文件动作必须显示路径、缩略图、替换按钮和打开位置按钮。
- 手动激发测试常驻，区分“只测试匹配”和“真实执行命中规则”。

### 运行监控

- 默认展示当前状态、最近回复、异常队列和 Webhook 队列。
- 单条日志打开右侧详情，显示规则命中、AI Thinking、判断库查询、发送动作、失败原因。
- 正常技术状态默认折叠；异常自动展开并给出修复入口。

### 系统设置

- 首次初始化只在缺配置时自动出现。
- 初始化后入口放在 `系统设置 > 初始化与修复`。
- API、Webhook、判断库、悬浮窗、开机启动、防休眠等低频配置按分区折叠。
- 危险区保留“彻底退出程序”，必须二次确认。

## 悬浮窗规范

悬浮窗只显示状态，不承担复杂配置。

| 模式 | 内容 |
| --- | --- |
| 展开态 | 当前状态、AI、本地服务、脚本、登录、实时钟、打开控制台、暂停/开启 Bot |
| 最小化态 | 状态灯、短状态、摘要、打开控制台、展开、隐藏 |

要求：

- 展开态固定尺寸，不允许无级缩放。
- 最小化态保留三个按钮：打开控制台、展开、隐藏。
- 悬浮窗关闭只是隐藏，可从主控台、Dock 或托盘恢复。
- 主控台和悬浮窗读取同一状态快照，标签、颜色、顺序和更新时间一致。

## 视觉规则

- 使用浅色主调：暖白背景、白色面板、珊瑚主操作色、语义状态色。
- 卡片圆角 8px，不做卡片套卡片。
- 状态不是按钮，操作才是按钮。
- 图标必须配文字；复杂说明放 tooltip 或右侧说明面板。
- 不使用大面积深色背景、装饰渐变、营销式 Hero。
- 表格、列表、状态行优先，适合长时间值守。

## 数据和接口

保持现有 IPC 兼容，后续可以新增只读聚合接口：

```text
main-get-workbench-snapshot
window.mainShell.getWorkbenchSnapshot()
```

建议快照包含：

- `status`
- `settingsHealth`
- `recordsSummary`
- `judgmentStatus`
- `notifyOutbox`
- `healthIssues`
- `lastTrace`
- `currentView`

前端再把快照转成 ViewModel：`statusLamps`、`ruleItems`、`logItems`、`settingSections`、`detailPanelItem`。

## 验收标准

- 主控台入口不重复、不遗漏。
- 用户 5 秒内能看到当前状态、异常和下一步动作。
- 客服页映射不遮挡 Top Bar 和全局操作。
- 悬浮窗展开态和最小化态均无遮挡、无文字截断。
- 规则、日志、异常、判断库和 Webhook 都能打开详情。
- `1440x900`、`1280x800`、`390x844` 无横向溢出。
- `test:status-ui`、`test:regressions`、`test:lifecycle`、`doctor` 通过。
